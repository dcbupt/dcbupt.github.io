G-VNFM

接收和处理来自NFVO的消息机制

RabbitConfiguration类中配置接收来自NFVO的消息队列、消息监听适配器对象绑定消息处理方法、消息监听容器关联消息队列与消息监听适配器。这样NFVO发送给G-VNFM的消息就可以被G-VNFM异步接收并处理了。
G-VNFM应用对于收到来自NFVO的消息，交给AbstractVnfmSpringAmqp类的onAction方法处理。
onAction方法首先将从队列取出的json字符串反序列化得到NFVMessage对象，然后调用父类AbstractVnfm的onAction方法处理消息，这种处理方式将通信方式与消息处理解耦，AbstractVnfm的多个子类对应多个不同的通信方式，它们都重载了onAction方法，因为不同通信方式接收消息的方式不同，但最后都会将消息转化为NFVMessage对象交给基类AbstractVnfm去处理。
AbstractVnfm的onAction方法会读取NFVMessage的action属性，action属性代表消息对象对应的网元生命周期事件。NFVMessage的不同action会执行不同的VNFM侧事件处理流程。

处理NFVO发送的INSTANTIATE消息

1、获取扩展参数并保存到对象属性中（参数用于与NFVO通过RABBITMQ通信）
2、解析VNFD等参数，生成VNFR
3、发送GRANT_OPERATION消息
4、如果收到返回的GRANT_OPERATION消息，发送ALLOCATE_RESOURCE消息
5、VNF的VNFCs的EMS执行INSTANTIATE EVENT对应的脚本
6、发送INSTANTIATE消息给NFVO

处理NFVO发送的MODIFY消息

执行VNF的CONFIGURE脚本。需要用到依赖其他VNFs的参数，通过VNFRecordDependency对象获取。
最后发送MODIFY消息给NFVO

处理NFVO发送的START消息

请求EMS执行START事件对应的脚本，返回START消息给NFVO

处理NFVO发送的HEAL消息

从NFVO发送的自愈请求消息中取出自愈原因，如果是switchToStandby，备份VNFC（自愈请求消息携带）执行START事件
如果不是switchToStandby，VNFC（入参）执行自愈事件

处理NFVO发送的SCALE_OUT消息

1、发送Scaling消息给NFVO，VNFM需要提供创建VNFC所需要的执行脚本给NFVO
2、如果收到NFVO返回的SCALED消息，取出新增VNFC对象，如果mode（NFVO发送SCALE_OUT消息时携带，表示创建的VNFC是否是备份VNFC）为STANDBY表明是备份VNFC，则设置VNFC对象的STATUS为STANDBY。
3、VNFC执行启动脚本，包括：
3.1、指定VNFC执行INSTANTIATE脚本
3.2、如果VNF依赖对象存在，VNFC接着执行CONFIGURE脚本
3.3、如果VNFC的STATE不是STANDBY，执行START脚本
4、发送SCALED消息给NFVO

处理NFVO发送的SCALE_IN消息

从消息对象中取出TARGET VNFR和待删除的VNFC
对于TARGET VNF的每个VNFC，请求EMS执行SCALE_IN事件对应的脚本，携带的参数中包括VNF静态参数、VNFC动态参数，同时增加待删除VNFC的动态参数（ip\fip\hostname），key以’removing_`开头

处理NFVO发送的RELEASE_RESOURCES消息

VNFR的所有VNFC执行TERMINATE事件包含的脚本
返回RELEASE_RESOURCES消息

NFVO

事件分发机制

Endpoint到NFVO进行事件注册：
RabbitConfiguration中配置了事件注册队列，支持其他应用（Endpoint）注册监听的事件到NFVO。
EventDispatcher实现了Endpoint注册事件到NFVO相关的几个方法
当从RabbitMQ队列收到其他Endpoint发送的注册事件消息时，会调用EventDispatcher的register方法完成事件注册
该方法将消息字符串反序列化得到封装的EventEndpoint类型，然后持久化到数据库。
EventEndpoint对象封装了Endpoint的名称、类型（rabbit\rest）、监听事件类型、projectId、NSRId、VNFRId等数据

NFVO分发事件给已经注册该事件的Endpoint：
EventNFVO作为Spring事件对象，继承自ApplicationEvent。封装了ApplicationEventNFVO对象，该对象封装实际的事件数据，包括事件类型枚举对象和序列化后的事件数据
EventDispatcher类实现ApplicationListener接口，监听Spring事件对象，在onApplicationEvent方法中实现事件对象的分发。
分发事件对象时，会从数据库中取出已经注册过的EventEndpoint对象，该对象封装了Endpoint要监听的事件类型。如果事件类型与此次分发的事件类型匹配且分发事件主体VNF与Endpoint要监听的VNF匹配，通过RabbitMQ发送消息给该Endpoint，携带EventNFVO封装的事件数据。
AbstractTask实现ApplicationEventPublisherAware接口，注入ApplicationEventPublisher对象，通过调用该对象的publishEvent方法触发所有监听该事件对象的监听器，执行监听器的事件分发方法。

接收VNFM发送的action事件消息

RabbitConfiguration配置类为接收来自VNFM的消息进行了以下配置：
接收VNFM消息的队列对象，name为vnfm.nfvo.actions.reply
对应多个队列的exchange对象，name为openbaton-exchange
Binding对象将接收VNFM消息的队列与exchange进行绑定，指定routingKey
MessageListenerAdapter监听器对象，该对象绑定消费消息的方法RabbitVnfmReceiver.actionFinished，即处理VNFM消息的方法
SimpleMessageListenerContainer容器，注入监听器对象和队列，用于将监听器对象和队列关联在一起。容器接收来自注入队列的消息，调用监听器对象绑定的消费消息方法。

处理来自VNFM的action消息

RabbitVnfmReceiver.actionFinished方法负责处理来自VNFM的消息
1、将String字符串反序列化为NFVMessage对象
2、调用VnfmManager对象的executeAction方法处理VNFM的action消息
3、根据Action类型得到对应AbstractTask子类，由该类负责处理，AbstractTask类实现Callable接口，线程池分配线程执行call方法
4、AbstractTask是抽象类，call方法的实现中会调用具体的方法以及一些抽象方法，不同子类对抽象方法的实现不同，以此来对不同类型消息进行不同的处理。

处理VNFM发送的GRANT_OPERATION消息

部署NS

NFVO侧发起实例化NS的VNFs请求给G-VNFM

RestNetworkServiceRecord控制器类处理NFVO DASHBOARD发起的REST请求，包括部署NS
部署NS的REST请求会交给控制器类的create方法处理
create方法会从HTTP BODY中取出json字符串，反序列化得到json对象，取出从前端传入的部署NS需要的参数（NSD、projectId、vduVimInstances、configurations），作为入参调用NetworkServiceRecordManagement类的onBoard方法，其中vduVimInstances、configurations参数可选
onBoard方法会将可选参数封装到DeployNSRBody对象中，然后调用deployNSR方法
deployNSR方法做如下事情：
1、根据projectId到数据库中更新对应VIM实例的状态，包含VIM实例的连接状态、镜像/网络/部署规格信息，然后持久化到数据库
2、判断：NSD定义的每个VNF对应的VNFM都已经注册给NFVO且可达
3、创建NSR
4、保证部署VDU的VIMs的可用性（类型、镜像、网络符合部署条件）
5、检测部署VNF的VIMs是否资源充足（虚拟CPU、虚机数目、内存、浮动IP）
6、创建描述NS的VNFs间的依赖关系的nsr.vnfRecordDependency对象
7、持久化NSR对象
8、前端入参保存的VNF配置参数k:v值 set 到 nsd.vnfd.configurations
最后调用VnfmManager的deploy方法，该方法做如下事情：
1、找到部署VDU的每一个VIM的实例对象并保存
2、保存扩展参数（与VNF本身无关的参数,包括VNFM需要用来与NFVO通信的RABBITMQ参数）
3、构建INSTANTIATE请求消息（包含vnfd、vnfd.df、vnfd.name、nsr.vlr、扩展参数、vim实例、vnf包对象）发送给VNFM

处理VNFM发送的GRANT_OPERATION消息

VnfmManager类的executeAction处理消息
首先根据message的action名得到Spring上下文创建好的Bean，这里是GrantoperationTask bean
该task类也是继承AbstractTask类，父类实现callable接口
AbstractTask的call方法会调用doWork方法，该方法由其子类实现。所以最后会调用GrantoperationTask类的doWork方法，该方法功能如下：
1、更新VNFR历史事件
2、NFVO持久化VNFR
3.1、如果不检查资源，从VNFR的VDU的可选VIMINSTANCENAME中随机指定负责部署VDU的VIM
3.2、如果检查资源，为每个vdu挑选一个vim实例加入map，如果存在vdu没有对应的vim实例，发送ERROR消息给VNFM
可以看到，授权资源做的操作就是检测每个VDU是否有可用于部署的VIM实例，如果满足，则挑选一个用于部署VDU的VIM实例，并将结果封装到消息对象后发送给VNFM

处理VNFM发送的ALLOCATE_RESOURCE消息

AllocateResourceTask的doWork实现NFVO侧处理分配资源消息的业务逻辑，主要包括：
1、创建VNFC，将得到的VNFC实例对象set到vnfr.vdu里
2、更新vnfr.historyLifecycleEvent
3、持久化VNFR对象
4、返回generic消息，携带更新后的vnfr

创建VNFC步骤：
1、从VIM实例的镜像中选择部署VDU的镜像
2、获取VNFC的CP
3、获取部署VNFC的flavourKey，优先考虑vdu.computation_requirement
4、动态生成VNFC的主机名
5、通过RABBITMQ消息队列发送RPC请求，请求VIM插件创建VNFC
6、取出Server对象的属性，创建VNFC实例对象

处理VNFM发送的INSTANTIATE消息

运行在VNFC的EMS执行完INSTANTIATE事件对应的脚本后，VNFM发送INSTANTIATE消息给NFVO，表示VNF的INSTANTIATE已经完成
NFVO处理VNFM发送的INSTANTIATE消息主要做以下事情：
1、NS中所有依赖该VNF的TARGET VNF对应的vnfRecordDependency更新依赖该VNF的参数
2、如果依赖VNF的TARGET VNF，其依赖的所有SOURCE VNF都已经INSTANTIATE完毕，发送MODIFY消息给VNFM，TARGET VNF执行MODIFY脚本
3、如果VNF依赖的其他VNFs都已经INSTANTIATE,VNF执行MODIFY事件。如果其依赖的VNFs还有未INSTANTIATE的，将VNF加入等待队列dependencyQueue
4.1、如果是按序执行NS下VNFRs的START事件
当VNF不依赖其他VNFs时，将STATE设为INACTIVE，当NS下所有VNFRs的STATUS都为INACTIVE时，依赖其他VNFs数最少的VNFR优先执行START事件
4.2、如果不按顺序执行NS下的VNFRs的START事件
当VNF不依赖其他VNFs时，执行START事件
5、如果NSR的所有VNFR都已经运行（状态为START），更新NSR的TASK为上载完成onboarded，并发布NSR实例化完成事件给监听该事件的其他系统

处理VNFM发送的MODIFY消息

如果按序执行START，当所有VNFR都处于INACTIVE状态（都已经执行过MODIFY事件），按依赖其他VNF数最少的VNFR优先执行START
不按顺序时，该VNFR接着执行START事件

处理VNFM发送的START消息

如果NSR所有的VNFRs按序执行START事件，取出当前未执行START事件的VNFR里有依赖其他VNF最少的VNFR，执行START事件（这时不需要判断是否NSR的所有VNFR都处于INACTIVE状态，因为收到VNFM的START消息说明已经有VNFR执行过START事件，也就说明之前已经判断了NSR的所有VNFR都处于INACTIVE状态）
否则返回NULL，不做任何处理

发起VNFC自愈流程

RestNetworkServiceRecord的REST接口：switchToRendundantVNFCInstance提供VNFC自愈服务
调用该接口的请求URL路径里指定nsr\vnfr\vdu的id，用于定位到VDU，再通过VNFC的STATE找到备份VNFC（备份VNFC的STATE为standBy），用于替换已经失效的VNFC。失效的VNFC对象封装在REST消息体里。
NFVO侧会将失效的VNFC的状态设置为FAILED，然后发送HEAL消息给VNFM，携带备份VNFC对象

发起扩容流程

RestNetworkServiceRecord的REST接口：postVNFCInstance用于往指定vdu里添加VNFC，其中nsrId\vnfrId\vduId都通过REST请求的路径获取，创建VNFC需要的参数：网络参数VNFComponent、部署节点VimInstanceNames通过消息体取出。
NSRManagement的addVNFCInstance函数用于具体执行创建流程。
1、通过nsrId到数据库取出NSR对象，校验projectId与NSR所属projectId是否一致。然后取出VNFR对象，再取出VDU对象。
2、判断VDU可以支持的扩容后VNFC最大数目与当前VNFC数目是否相等，相等则抛出异常， 不受理创建VNFC请求。
3、校验是否创建VNFC提供的网络参数CP所关联的VLR属于VNFR的VLR集合里
4、将网络参数VNFComponent对象添加到VDU维护的VNFC网络参数集合里
5、获取描述VNF依赖参数的对象VNFRecordDependency、VNF包对象，随机从VimInstanceNames里选取一个Vim实例对象
6、构造SCALE_OUT消息，发送给VNFR所属的VNFM

处理VNFM发送的SCALING消息

ScalingTask的doWork函数处理VNFM发送的SCALING消息
1、找到将要创建的VNFC实例对应的VNFComponent和所属的VDU
2、选取部署VNFC的VIM并创建VNFC
2.1、判断部署VNF的VIM实例是否有足够的资源时创建VNFC
2.1.1、部署新增VNFC的VIM实例没有选定，为VDU选择第一个满足部署条件的VIM实例
2.1.2、部署新增VNFC的VIM实例已经选定
2.2、不判断部署VNF的VIM实例是否有足够资源时创建VNFC
2.2.1、部署新增VNFC的VIM实例没有选定，随机选择一个VIM实例
2.2.2、部署新增VNFC的VIM实例已经选定
3、将执行的VNF事件记录保存到vnfr
4、持久化VNFR
5、返回SCALED消息

处理VNFM发送的SCALED消息

1、针对每个依赖VNF的TARGET VNF，创建VNFRecordDependency对象，封装VNF提供的静态参数和新增VNFC参数
2、新增VNFC的动态参数更新到nsr.vnfRecordDependency对象中
3、发送MODIFY消息给VNFM，携带新创建的VNFRecordDependency对象
4、发送NSR级别的SCALE_OUT消息给监听该NSR的其他系统
5、发送VNFR级别的SCALED消息给监听该

发起缩容流程

REST的deleteVNFCInstance接口用于处理缩容请求，传入nsrId\vnfrId\vduId\vnfcId
1、找到所有依赖待删除VNFC的VNFRecordDependency，针对每个TARGET VNF，移除对该VNFC参数的依赖。
1.1、发送SCALE_IN消息给VNFM，携带TARGET VNF的VNFR，待删除VNFC
1.2、TARGET VNF的VNFRecordDependency对象删除保存的待删除VNFC参数
2、 通过RPC请求VIM插件执行删除VNFC操作，请求参数携带VNFCId、VimInstanceId
3、VNFR对象中删除VNFC虚机的相关参数
3.1、VDU对象的VNFComponent属性删除VNFC对应的网络参数，vnfc_instance属性删除VNFC实例
3.2、VNFR的vnf_address属性（保存VNFCs的IP）删除VNFC的IP
4、发送SCALE_IN完成消息给监听该VNFR的SCALE_IN事件的Endpoint（这里是VNFM）
5、持久化更新后的NSR对象

发起终止流程

RESTFUL的delete方法删除NSR，这会触发所有VNFR的终止流程
对于每一个VNFR，NFVO发送RELEASE_RESOURCE消息给VNFM

处理VNFM发送的RELEASE_RESOURCES消息

通过RPC调用VIM插件，删除VNF的所有VNFC

完
