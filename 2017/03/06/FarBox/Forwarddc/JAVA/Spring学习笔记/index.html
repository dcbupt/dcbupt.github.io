正确理解依赖注入DI

作为Spring的两大核心：DI和AOP，DI可以说是你使用Spring框架必备的基础。

注入的是对象的引用

通过依赖注入，你可以在一个类中将另一个类对象的引用作为自己的属性。

这里说的是类对象的引用！假如你在类的构造方法中new了另一个对象，那就称不上是依赖注入了。

为什么不new呢？

这又和DI的另一个特点有关：
依赖注入通常用于注入一个接口引用。这么做是为了让接口的实现类与使用依赖注入的类解耦。
怎么理解？举个例子：
类A通过DI注入了接口B的引用，而类C实现了接口B。假如某一天，我不想用类C作为实现了，那我只需要 创建接口B的实现类D即可。

而替换实现类对类A来说，是不可见的！ 这就做到了类A与接口B的实现类的解耦。

假如你是直接在类A中new类C，那么如果替换实现，类A的代码也需要修改，这显然是不合理的。
其实，在另一个类中new一个对象的方式，与硬编码无异，称为硬初始化，而DI就是避免硬初始化。

使用DI

POJO->BEAN，浅谈Spring装配

Spring中使用的每一个类，都需要被装配成BEAN。DI中注入的对象引用，该对象必须是一个BEAN，而被注入的类也必须是一个BEAN。
前面提到，DI通常注入的是接口引用，其实接口不必装配，但接口的实现必须要装配成BEAN。

BEAN可以理解为Spring认可的JAVA类。通过几种装配方式，可以让简单JAVA类被Spring框架所识别。

下面介绍三种Spring装配方式：

扫描

  • 通过@Component告诉Spring，该类可以被扫描为BEAN

  • 接口不需要添加@Component，只需要为其实现类添加即可。而对外依然可使用接口，Spring会找到实现了该接口的BEAN

  • 通过@Component与@ComponentScan创建扫描类启动扫描功能

    Spring会扫描CDPlayerConfig包及其子包下的@Component，@ComponentScan还可以指定包名

创建配置类

  • 如果你看到一个类，用@Configuration修饰而不包含@ComponentScan，表明它是一个JAVA配置类
  • 该类可以通过声明一些方法的方式创建BEAN，方法名和返回类型都为类名，注意方法名要小写
  • 需要特别强调，方法的返回类型可以是一个接口， 在Spring项目中使用该接口时，实际使用的是其实现类

    xml方式

  • 通过标签创建BEAN，注意其对构造参数和属性参数的注入方式
    • c命名方式创建带构造参数的BEAN
      引用参数 多参数使用下标
    • contructor方式创建带构造参数的BEAN
      引用参数 多个参数（包含列表）
    • p命名方式创建带属性的BEAN
      引用参数 多个参数（包含列表）
    • Property方式创建带属性的BEAN
      引用参数 多个参数（包含列表）

      混和装配

  • 配置类中引入其他配置类、xml配置文件
  • xml配置文件中引入三种方式的装配文件

SpringMVC

MVC我理解就是一种服务器端的设计模式。而SpringMVC就是使用Spring框架实现的MVC设计模式。
Model–>对应业务数据模型
View–>对应反馈给用户的页面，通常会渲染Model数据
C–>对应服务器端的控制器，用于实现业务逻辑，是数据模型与视图之间的桥梁。

SpringMVC的一般流程

①客户端请求
②DispatcherServlet根据URL选择Controller
③Controller调用业务层接口实现业务逻辑
④返回Model对象与逻辑视图名
⑤ViewResolver解析到视图的物理地址
⑥View渲染Model数据
⑦返回给客户

业务类POJO

业务类POJO在MVC中是怎么用的？

业务类POJO用于封装与用户交互产生的数据

举个最简单的例子，比如一个提交表单的视图，表单提交后，控制器会有对应的方法处理POST请求，方法的参数中会包含业务类对象，该对象就封装了POST参数。即，将用户提交的表单数据用JAVA业务类描述
对于业务类对象的属性，Spring提供了对这些属性的一些校验注解来限制属性的值。

  • 校验注解举例
    • @NotNull && @Size(min=5, max=16, message="{username.size}")限制属性的值非空，且长度在5~16字符之间，如果POST参数不满足条件，视图中可以获取message作为错误提示信息

message使用了属性占位符，即通过定义在类路径下的key-value文件获取实际的value。值得一提的是，该文件也可以引用自己的占位符， 例如{min}，对应校验注解的参数名。
如何触发校验？需要使用Spring视图技术的表单绑定功能。

视图技术的表单绑定功能允许其指定业务类POJO，并将表单参数与POJO参数绑定在一起

在重定向中携带请求数据

当控制器方法处理完请求时，会指定视图作为响应，而视图可以访问Model中的数据并渲染到页面上。
但，当控制器方法处理的是POST请求时，最佳实践是处理完毕后重定向到另一个URL上（对应一个控制器方法）而非视图上

这样可以防止客户端在做刷新或者后退操作时，重复发起可怕的POST请求

但，重定向不同于返回视图。
返回视图时，视图和控制器方法针对的是同一请求，所以视图可以获取控制器方法传递的Model中的数据
然而重定向是对另一个URL的请求，会调用另一个控制器方法，是一个新的请求，不会携带之前的数据
如何在重定向中携带数据呢？

  • 使用URL模板

    • URL模板适用于数据作为重定向URL的一部分
    • URL模板中使用占位符来指代数据，占位符的源是Model中的key-value对
    • 如果Model保存的数据不在URL模板的占位符里，那么将作为请求参数携带到重定向的URL中
    • 无论是作为占位符还是请求参数，URL模板只能用于携带简单数据（数字或字符串）

  • 发送业务类对象
    既然URL模板只能向重定向传递简单数据，那如何实现传递业务类对象呢？
    一种思路是：
    将需要传递给重定向的业务类对象放到会话中（是否可以理解为将业务类对象设为会话作用域？）
    Spring不需要我们这么做，它对这一操作做了封装，我们只需要：

Thymeleaf

Thymeleaf的出现是希望取代JSP，JSP有以下缺点：

  • JSP并非是HTML，虽然JSP最终会被渲染成HTML，但浏览器很难将JSP页面展现的效果在审美上与使用HTML一致

    言下之意，对JSP标签的渲染常常出现问题，使得渲染的效果无法与原生的HTML一致

  • JSP与Servlet紧耦合，所以JSP无法用于不基于Servlet的WEB应用

JSP其实是取代了HTML，JSP有一套自己的标签库，与HTML标签是完全独立的，所以JSP需要基于Servlet来把自己的标签库渲染为HTML

Thymeleaf的优势：

  • 直接用于HTML页面，对HTML页面渲染

    Thymeleaf标签用作HTML标签的属性，属性值使用Thymeleaf表达式动态计算，所以其本身就依附于HTML

Thymeleaf实际上是渲染HTML标签的属性（动态计算属性值），所以其可以嵌入HTML原生页面，这一点是JSP做不到的

  • 逻辑视图名到Thymeleaf的映射

Spring Web Flow

当WEB应用可以按照固定的流程走时（例如购物应用，分为”注册-订购-支付“等流程），Spring提供了SWF框架供我们开发此类应用
使用框架的第一步就是如何在项目中安装它，所以先介绍如何安装SWF

SWF在项目中的安装

所谓安装SWF，就是创建一些SWF需要的BEAN，下面逐个介绍

  • flow-executor
    其地位相当于SWF的”根节点”，executor会在用户每次进入流程时创建流程，即驱动流程执行
  • flow-registry
    负责加载流程定义
    注意：每个流程通过id来标识，指定base-path加载的流程，其id为相对于base-path的路径
  • FlowHandlerMapping
    将URL请求map到flow-registry定义的流程中
    如何map：通过URL的相对路径作为流程id，匹配flow-registry中对应的那个流程
    乍一看，这不是MVC模式中DispatcherServlet做的事情吗？但对于SWF，是通过FlowHandlerMapping帮助DispatcherServlet做这件事

流程三要素：状态、转移、流程数据。
可以将流程想象成一个状态转移图，对应上面的流程三要素。

状态

状态是流程内部中转时的节点，在到达状态（节点）时会做一些事情

下面对SWF中定义的状态作介绍：

  • 视图状态
    顾名思义，流程执行到视图状态时返回视图，如果视图中需要将表单数据绑定给POJO，设置model属性

  • 行为状态
    行为状态会去调用Bean的方法，并根据方法执行的结果转移到另一个状态

  • 决策状态
    决策状态会根据表达式返回的boolean结果不同，执行不同的流程

  • 子流程状态
    执行到子流程状态，会触发另一个状态，并根据子流程的结束状态决定下一个状态

  • 结束状态
    结束状态分为：一般流程和子流程的结束状态
    对于子流程的结束状态，会用于返回给调用流程
    注意：结束状态可能不止一个！

转移

转移的作用是根据触发的事件实现状态间的转移

转移用于视图、行为、子流程状态

  • 对于视图状态，事件是用户的动作
  • 对于行为视图，事件是表达式的结果
  • 对于子流程，事件是子流程结束状态ID
  • 全局转移

    如果转移适用于所有的状态，可以将其定义为全局转移，全局转移会默认应用于所有的状态
    使用global-transitions标签定义全局转移

DAO && 持久化框架

-DAO
为了避免持久化的逻辑分散到应用的各个组件中，最好将数据访问的功能放到一个或多个专注于此项任务的组件中，这样的组件称为DAO（数据访问对象）或Repository

业务对象本身并不会处理数据访问，而是将数据访问委托给Repository接口，Repository接口确保其与业务对象的松耦合。

  • 持久化层
    数据库访问有不同的框架，可以通过JDBC也可以通过ORM框架，所以将访问底层数据库的框架抽象为持久化层，例如：JDBC\Hibernate\MyBatis

DAO可以理解为一个提供了访问数据库方法的接口，DAO接口的实现类依靠持久化层访问数据库

Spring将DAO层（Repository）又细化为两个部分：模板和回调。
模板负责访问数据库中的固定部分（准备资源、开始事务、提交/回滚事务、关闭资源和异常处理）
回调是自定义的数据访问代码：处理事务、返回数据

DAO层的Repository BEAN会注入持久化框架的BEAN，即依赖持久化框架访问数据库。
DAO层的模板部分由持久化框架保证，而回调部分则是在DAO层由程序员实现（即SQL语句等，作为持久化框架API的参数）。

Spring支持多个持久化框架。

Spring消息

异步消息

异步消息有两个主要概念：消息代理和目的地。
当一个应用发送消息时，会将消息交给一个消息代理。消息代理确保消息被投递到指定目的地，同时解放发送者，使其继续其他的业务。
不同消息系统提供不同消息路由模式，但有两种通用目的地：队列queue和主题topic。队列queue对应点对点模型，主题topic对应发布/订阅模型。

点对点消息模型

消息代理将消息放到队列中，接收者请求消息时，消息会从队列取出，投递给接收者，然后将消息从队列中删除，保证消息只被一个接收者消费。
可以有多个接收者消费消息。

发布/订阅消息模型

消息会发送给一个主题，多个接收者可以监听同一个主题。消息不再只投递给一个接收者，而是主题的所有订阅者都会接受此消息的副本。

同步通信的缺点

  • 需要等待响应
  • 客户端接口与远程服务接口耦合
  • 客户端与远程服务的位置耦合
  • 客户端与远程服务的可用性耦合

消息监听容器包含多个消息监听器，监听器容器从队列/主题获取消息，分发给任意一个对此消息感兴趣的消息监听器。

建立RPC服务

定义接口和实现类后，将实现类装载为Bean，注入到JmsInvokerServiceExporter，该接口即可作为基于JMS的服务。
JmsInvokerServiceExporter可以充当JMS监听器，底层也是使用Spring异步消息机制来对外提供服务。

JmsInvokerProxyFactoryBean代理了通过JmsInvokerServiceExporter所导出的JMS服务

JMS一直是JAVA应用中主流的消息解决方案，直到高级消息队列协议AMQP的出现。

AMQP

AMQP为消息定义了wire-level protocol，规范了消息格式，消息在生产者和消费者之间传递的时候都会遵循这个格式。因此相比JMS，可以跨不同AMQP的实现、跨语言、跨平台。

生产者不会直接将消息发送到队列，而是发到一个Exchange。Exchange会绑定到一个或多个队列上，它负责将信息路由到队列上。

AMQP定义了四种不同类型的Exchange（DIRECT\TOPIC\HEADER\FANOUT），每种都有不同的路由算法，但核心都是用消息的routing key或参数与Exchange与队列之间绑定的routing key进行匹配。

RabbitMQ

RabbitMQ是一个流行的开源消息代理，实现了AMQP。Spring AMQP为RabbitMQ提供了支持，包括RabbitMQ连接工厂、模板以及Spring配置命名空间。

配置RabbitMQ

使用Spring为RabbitMQ提供的命名空间装配连接工厂，会默认监听服务器的5672端口，用户名密码均为guest
装配队列、Exchange、binding、routing key

RabbitTemplate

对应JmsTemplate，RabbitTemplate用于基于RabbitMQ发送和接收消息的场景。装配该BEAN也可以使用Spring为RabbitMQ提供的命名空间

发送消息时，调用convertAndSend方法，该方法有多个重载，可以传入Exchange名、routing key、消息对象
Exchange名和routing key可以有默认值，这时需要在RabbitTemplate装载的时候配置

接收消息时，调用receive或receiveAndConvert方法不会同步阻塞，如果没有消息，会立即返回null

消息驱动的AMQP与消息驱动的JMS类似，Spring提供了良好的支持，我们只需要负责编写消费者类，配置到监听器，最后加入监听器容器即可。

REST API

REST基本概念

当谈论 REST 时，有一种常见的错误就是将其视为 “ 基于 URL 的 Web 服务 ”—— 将 REST 作为另一种类型的远程过程调用（ remote procedure call ， RPC ）机制
恰好相反， REST 与 RPC 几乎没有任何关系。 RPC 是面向服务的，并关注于行为和动作；而 REST 是面向资源的，强调描述应用程序的事物和名词
为了理解 REST 是什么，我们将它的首字母缩写拆分为不同的构成部分：
表述性（ Representational ）： REST 资源实际上可以用各种形式来进行表述，包括 XML 、 JSON （ JavaScript Object Notation ）甚至 HTML —— 最适合资源使用者的任意形式；
状态（ State ）：当使用 REST 的时候，我们更关注资源的状态而不是对资源采取的行为；
转移（ Transfer ）： REST 涉及到转移资源数据，它以某种表述性形式从一个应用转移到另一个应用。
更简洁地讲， REST 就是将资源的状态以最适合客户端或服务端的形式从服务器端转移到客户端（或者反过来）
在 REST 中，资源通过 URL 进行识别和定位。
REST 中会有行为，它们是通过 HTTP 方法来定义的。具体来讲，也就是 GET 、 POST 、 PUT 、 DELETE 、 PATCH 以及其他的 HTTP 方法构成了 REST 中的动作

表述转换

控制器以 Java 对象的方式来处理资源。控制器完成了它的工作之后，资源才会被转化成最适合客户端的形式。
Spring 提供了两种方法将资源的 Java 表述形式转换为发送给客户端的表述形式：
内容协商（ Content negotiation ）：选择一个视图，它能够将模型渲染为呈现给客户端的表述形式；
消息转换器（ Message conversion ）：通过一个消息转换器将控制器所返回的对象转换为呈现给客户端的表述形式。

内容协商

Spring 的 ContentNegotiatingViewResolver 是一个特殊的视图解析器，它考虑到了客户端所需要的内容类型。
要理解 ContentNegotiating-ViewResolver 是如何工作的，这涉及内容协商的两个步骤：
1、确定请求的媒体类型；
2、找到适合请求媒体类型的最佳视图。

ContentNegotiatingViewResolver 将会考虑到Accept头部信息并使用它所请求的媒体类型，但是它会首先查看URL的文件扩展名。如果URL在结尾处有文件扩展名的话， ContentNegotiatingViewResolver 将会基于该扩展名确定所需的类型
最后，如果没有Accept头部信息，并且扩展名也无法提供帮助的话， ContentNegotiatingViewResolver 将会使用“/”作为默认的内容类型，这就意味着客户端必须要接收服务器发送的任何形式的表述
ContentNegotiatingViewResolver 要求其他的视图解析器将逻辑视图名解析为视图。解析得到的每个视图都会放到一个列表中。这个列表装配完成后，ContentNegotiatingViewResolver 会循环客户端请求的所有媒体类型，在候选的视图中查找能够产生对应内容类型的视图。第一个匹配的视图会用来渲染模型

从Spring 3.2开始， Content-NegotiatingViewResolver 的大多数Setter方法都废弃了，鼓励通过 Content-NegotiationManager 来进行配置，借助 Content-NegotiationManager 我们所能做到的事情如下所示：
• 指定默认的内容类型，如果根据请求无法得到内容类型的话，将会使用默认值；
• 通过请求参数指定内容类型；
• 忽视请求的Accept头部信息；
• 将请求的扩展名映射为特定的媒体类型；
• 将 JAF（Java Activation Framework）作为根据扩展名查找媒体类型的备用方案。
配置 ContentNegotiationManager 的方法：
• 通过 ContentNegotiationManagerFactoryBean 间接创建bean；
• 重载 WebMvcConfigurerAdapter 的 configureContentNegotiation() 方法。
一般而言，如果我们使用XML配置 ContentNegotiationManager 的话，那最有用的将会是ContentNegotiationManagerFactoryBean
如果使用Java配置的话，获得 ContentNegotiationManager 的最简便方法就是扩展 WebMvcConfigurerAdapter 并重载configureContentNegotiation() 方法
现在，我们已经有了 ContentNegotiationManagerbean ，接下来就需要将它注入到 ContentNegotiatingViewResolver的 contentNegotiationManager 属性中

ContentNegotiatingViewResolver 最大的优势在于，它在 Spring MVC 之上构建了 REST 资源表述层，控制器代码无需修改
如果面向人类用户的接口与面向非人类客户端的接口之间没有太多重叠的话，那么ContentNegotiatingViewResolver的优势就体现不出来了
ContentNegotiatingViewResolver 还有一个严重的限制。作为 ViewResolver 的实现，它只能决定资源该如何渲染到客户端，并没有涉及到客户端要发送什么样的表述给控制器使用
ContentNegotiatingViewResolver还有一个相关的小问题，所选中的View会渲染模型给客户端，而不是资源

消息转换器

当使用消息转换功能时， DispatcherServlet 不再需要那么麻烦地将模型数据传送到视图中。实际上，这里根本就没有模型，也没有视图，只有控制器产生的数据，以及消息转换器（message converter）转换数据之后所产生的资源表述
Spring自带了各种各样的转换器，所以要使用它们的话，不需要Spring配置。但是为了支持它们，你需要添加一些库到应用程序的类路径下。例如，如果你想使用 MappingJacksonHttpMessageConverter 来实现JSON消息和Java对象的互相转换，那么需要将 Jackson JSON Processor 库添加到类路径中。类似地，如果你想使用Jaxb2RootElementHttpMessageConverter 来实现XML消息和Java对象的互相转换，那么需要JAXB库。

如果使用了消息转换功能的话，我们需要告诉Spring跳过正常的模型/视图流程，并使用消息转换器。有不少方式都能做到这一点，但是最简单的方法是为控制器方法添加 @ResponseBody 注解
@ResponseBody注解会告知Spring，我们要将返回的对象作为资源发送给客户端，并将其转换为客户端可接受的表述形式。更具体地讲， DispatcherServlet 将会考虑到请求中Accept头部信息，并查找能够为客户端提供所需表述形式的消息转换器
谈及Accept头部信息， @RequestMapping 注解使用了produces属性表明这个方法只处理预期输出为JSON的请求

@ResponseBody能够告诉Spring在把数据发送给客户端的时候，要使用某一个消息器，与之类似，@RequestBody 也能告诉Spring查找一个消息转换器，Spring通过查看请求中的Content-Type头部信息，将来自客户端的资源表述转换为对象
@RequestMapping有一个consumes属性，consumes属性的工作方式类似于produces，不过它会关注请求的Content-Type头部信息。

如果在控制器类上使用@RestController来代替@Controller的话，Spring将会为该控制器的所有处理方法应用消息转换功能，不用每个控制器方法都加上@ResponseBody

作为@ResponseBody的替代方案，控制器方法可以返回一个ResponseEntity对象。ResponseEntity中可以包含响应相关的元数据（如头部信
息和状态码）以及要转换成资源表述的对象。
除了包含响应头信息、状态码以及负载以外，ResponseEntity还包含了@ResponseBody的语义，因此负载部分将会渲染到响应体中，就像之前在方法上使用@ResponseBody注解一样。如果返回ResponseEntity的话，那就没有必要在方法上使用@ResponseBody注解了

@ExceptionHandler注解能够用到控制器方法中，用来处理特定的异常。
这里，它表明如果在控制器的任意处理方法中抛出自定义的异常，就会调用异常处理方法来处理异常

@ResponseStatus(HttpStatus.NOT_FOUND)注解的方法会返回HTTP状态码，如果控制器方法每次返回的状态码唯一，可以使用这种方式。这时可以不返回ResponseEntity对象，直接使用@ResponseBody注解

当创建新资源的时候，将资源的URL放在响应的Location头部信息中，并返回给客户端是一种很好的方式。因此，我们需要有一种方式来填充响应头
部信息，此时我们的老朋友ResponseEntity就能提供帮助了
我们其实没有必要手动构建URL，Spring提供了UriComponentsBuilder，可以给我们一些帮助。为了使用UriComponentsBuilder，我们需要做的就是在处理器方法中将其作为一个参数
在处理器方法所得到的UriComponentsBuilder中，会预先配置已知的信息如host、端口以及Servlet内容。它会从处理器方法所对应的请求中获取这些
基础信息。基于这些信息，代码会通过设置路径的方式构建UriComponents其余的部分

REST客户端

通常来讲，移动或JavaScript应用会是REST API的客户端，但是Spring应用也完全可以使用这些资源

REST客户端使用RestTemplate调用服务端REST接口。RestTemplate定义了36个与REST资源交互的方法，这里面只有11个独立的方法，主要用到的就是对应HTTP请求的GET\POST\PUT\DELETE操作，其中有十个有三种重载形式，而第十一个则重载了六次，这样一共形成了36个方法。
大多数操作都以三种方法的形式进行了重载：
• 一个使用java.net.URI作为URL格式，不支持参数化URL；
• 一个使用String作为URL格式，并使用Map指明URL参数；
• 一个使用String作为URL格式，并使用可变参数列表指明URL参数
