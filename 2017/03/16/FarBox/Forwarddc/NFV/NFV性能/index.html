NFV性能

发表于 2017-03-16 | 更新于: 2017-03-16 | 分类于 NFV | 阅读次数:
MANO架构中的组件性能

VNFM性能要求

容量管理规格

VNFM的容量管理涉及VNFD、VNF、策略、VM、VIM

首先，VNFM何时会触及到这几项数据呢？

答案是：在VNF实例化过程中！
1、O->M下发VNFD、策略
2、M解析D后，生成VNF InstanceID和VM List，作为参数调用O的API请求资源池中的虚拟资源
3、O根据VM List计算出满足需求的VIM，将VIM ID发给M
4、M根据VIM ID，到指定的VIM中完成VNF虚拟资源的创建

根据上述流程，可以发现：

VNFD：

VNFD应该是文本文件，描述VNF的规格帮助M计算得到VM List，所以VNFM无需管理VNFD，只需要存储和解析VNFD，因此只和VNFM的存储空间有关。考虑到VNFD文件很小，所以其可管理的VNFD数量与性能无关。

VNF

VNFM在实例化过程中，会生成VNF的唯一标识VNF Instance，而该标识存在的意义是作为在接口中传递的参数，所以VNFM可管理的VNF不受限制。其维护的VNF Instance List对应所管理的VNF，也许其还要维护VNF Instance与VNFD、策略、VM List、VIM ID的对应关系，但这些都是静态数据，与性能无关。

策略，VM，VIM

承上，这三项数据也是和VNF Instance紧密联系的静态数据，因此与VNFM性能无关。

单VNF支持的最大VM数量

这在VNFM看来，VNF需要多少VM无所谓，其只需要负责维护VNF Instance与VM List的对应关系即可。

业务技术指标

VNFM服务启动时间：

作为VNFM的厂商，会将VNFM部署到服务器上，因此VNFM的启动时间需要结合两方面考量：

a、VNFM启动时间对业务的影响：
具体来说，就是M管理的VNF在M重启服务的这段时间，对业务造成中断所产生的损失
b、硬件服务器的性能

并发任务同时操作VM的数量

在VNF生命管理周期中，M到VIM的关于VM创建和删除的并发任务必然会发生，而并发数目与厂商对VNFM的实现紧耦合，高并发的创建VM带来的好处体现在快速部署以及扩缩容的快速响应。
另一方面，因为并发操作VM是M与VIM接口上的操作，所以还要看VIM是否支持并发创建VM以及在并发的压力下，创建VM的时延，这方面主要是考察VIM的性能。

并行实例化VNF最大数量

O可能调用M的多条实例化VNF请求API，此时M内部可能会涉及并发解析VNFD、生成VNF Instance ID、分配JobID等一系列任务，因此此项是对M性能的合理考察点。

并行弹性伸缩VNF最大数量

同上，扩缩容流程中也会涉及到M-O、M-VIM、M-VNF、M-EMS这几个接口之间的多次调用，因此其支持的并发扩缩容VNF数量也是M应该考察的性能。

单个VNF实例化时间

虽然不同VNF，其规格会有差异，这种差异势必造成VNF实例化时间上的差异，但仍有必要针对同一种VNF，列出其实例化所需要的时间指标，作为性能的考察点。因为不同VNF对实例化的时间要求，与其所扮演的角色直接相关，是可以规划的。

单个VNF弹性伸缩时间

同上。

VNF弹性伸缩决策响应时延

此时延与M获取VNF业务状态信息、VNF虚拟层状态信息有关
触发弹性伸缩条件的第一步是M获取到VNF的状态信息满足触发条件，之后根据策略文件执行VNF弹性伸缩。
所以，这可能与M获取VNF状态信息的方式有关，建议就这一层面展开调研。

VNFM客户端命令响应时间

M采取BS架构，此种架构不存在客户端（因为客户端只有浏览器，不做任何业务处理与数据存取），只需要通过浏览器访问服务器端的数据，服务器端负责完成业务逻辑和业务数据的存取。
此项指标应该就是通过浏览器

C5、C6、C10接口处理时延

接口粒度的时延性能。建议以功能粒度划分，不应该以接口为导向，应该以功能为导向，因为接口是为功能服务的。

小结

VNFM容量相关的指标，均为静态数据，只和存储容量有关，但私以为：
VNFM容量真正的意义在于，其能管理的VNF上限。因为真正占用M的虚拟资源的是高并发，当多个VNF发生实例化、扩缩容、终止操作时，都会依赖于M在多个接口上的调用。
所以，M的性能指标的关键词就是并发。

即业务技术指标中，并发情况下VNF生命周期管理的时延作为衡量VNFM性能的依据。具体可以：
1、同种VNF不同数量等级条件下，M的业务技术指标的测试、M占用虚拟资源情况的测试
2、实际中，M要管理不同种类的VNF时，需要针对具体场景，设计相关VNF的配比，再做业务技术指标测试和M占用虚拟资源情况的测试
