G-VNFM的实现

OBv3为VNFM提供了三种选择：

  • 使用G-VNFM或者JujuVNFM作为项目一部分
    这种方式能让你快速部署，将注意力集中在VNF包的实现上。两种VNFM都支持基于CSAR的VNF包
  • 使用JAVA SDK或者PYTHON SDK构建VNFM
    OB提供了JAVA SDK称作vnfm-sdk，这能帮助你为你自己的VNF简单创建一个VNFM。关于NFVO与VNFM之间的通信，可以使用vnfm-sdk-amqp或者是vnfm-sdk-rest
  • 使用你自己的VNFM来连接NFVO（通过REST APIS或AMQP）
    这种方式允许你整合你的VNFM和VNF到OB项目。这种方式下，O与M的通信需要通过REST接口。

G-VNFM

G-VNFM是根据ETST MANO规范的一种实现。

G-VNFM扮演O和VNFs中间件，特别是为了与装了VNF软件的虚机之间的交互。

为了完成VNF的生命周期管理，其与EMS交互，EMS扮演了虚机代理的角色，执行VNF包中的脚本或者执行通过VNFD指定链接的脚本。

可以认为，M对VNFs的生命周期管理，在VNF层面是通过运行在VM的EMS执行脚本完成的。

G-VNFM可以管理一个VNF实例、多个同类型或不同类型VNF实例

O-M-EMS之间的通信是基于RABBITMQ AMQP协议。

虚机的部署

基于ETSI MANO B.3，VNF的实例化流程可以通过两种方式实现：
1、由NFVO完成资源分配
2、由VNFM完成资源分配

G-VNFM采用第一种方式，向NFVO发送两条消息：
1、授权资源消息：
检测是否有足够的硬件资源部署VNF。如果收到了NFVO返回的此消息，说明资源充足，于是发送资源分配消息
2、分配资源消息：
此条消息要求NFVO去创建部署VNF需要的虚机

虚机创建之后，VNFRecord就被值填充了，例如ips，可以直接通过VirtualNetworkFunctionRecord→VirtualDeploymentUnit→VNFCInstance对象找到

脚本执行

每一个VNF生命周期管理接口的操作中，VNFM发送脚本给EMS，运行在虚机本地
注：脚本来自VNF包，你需要创建

脚本的顺序在NetworkServiceDescriptor→VirtualNetworkFunctionDescriptor→LifecycleEvents.中定义

可以看到，脚本与VNF生命周期事件是对应的，事件与脚本执行的关系如下：

  • INSTANTIATE
    安装VNF阶段执行脚本
  • CONFIGURE
    安装完后执行脚本（如果VNF依赖其他VNFs，需要执行脚本得到其他VNFs提供的参数，例如ip）
  • START
    在INSTANTIATE和CONFIGURE（如果存在）事件后执行脚本
  • STOP
    关闭VNF阶段执行
  • TERMINATE
    终止VNF阶段执行
  • SCALE_IN
    当虚机VNFC扩缩时执行脚本

脚本可获取的参数定义在了VNFD里，分为两类：

  • 配置参数
  • 脚本自动获取的参数（私有IP、浮动IP、主机名）

在INSTANTIATE事件的脚本中，会将这两类参数设为环境变量

在CONFIGURE脚本中，INSTANTIATE事件的参数依然可得，但还包括来自其他VNF源的环境变量，这些参数定义在其他VNF的VNFD的provide field里。而该VNF需要的参数定义在自己的VNFD的require field里。获取外部VNF变量后，这些环境变量名变为$type_of_vnf_source_name_of_parameter
CONFIGURE事件里执行的脚本需要用“VNF名_脚本名”的方式命名

虚机终止

STOP事件意味着仅仅是停止VNF服务，之后还会重启它。

TERMINATE事件从入网点删除虚拟资源（虚机），由NFVO发起和执行。在终止之前会运行特定脚本。

DUMMY VNFM：

仿制的VNFM全称是：DUMMY VNF MANAGER AMQP，其模仿真正的VNFM的行为，使用AMQP与NFVO通信。

如果你使用DUMMY VNFM部署一个网络服务，它不会创建任何的资源。当NFVO向DUMMY VNFM发送命令时，它会
给出回应，就好像它做了，尽管它什么都没做。通过这种方式可以测试OPENBATON而不用部署任何东西。

为了执行DUMMY VNFM，你需要运行并配置NFVO。

源码安装DUMMY VNFManager AMQP

git clone项目到/opt/openbaton

执行编译
cd /opt/openbaton/dummy-vnfm-amqp; ./dummy-vnfm.sh compile

使用DUMMY VNFManager AMQP

运行DUMMY VNFM
cd /opt/openbaton/dummy-vnfm-amqp;./dummy-vnfm.sh start

这样做会创建一个新的screen窗口，你可以使用screen -x openbaton进入

如果你想要运行DUMMY VNFM在其他主机上而不是运行着RABBITMQ的主机，你首先需要进行配置。

你可以通过项目资源路径的application.properties去配置，或者创建/etc/openbaton/dummy-vnfm-amqp.properties 并拷贝application.properties的内容进来后进行配置

配置的具体内容为：改变属性spring.rabbitmq.host和属性spring.rabbitmq.port的值，指定RABBITMQ运行的主机和端口。

如果你创建了/etc/openbaton/dummy-vnfm-amqp.properties，DUMMY VNFM只会使用此配置文件，所以需要确保application.properties文件的所有属性都在这个文件里

使用DUMMY VNFM部署NS，你需要在NFVO中存储一个TEST类型的VIM实例以及一个VNFD文件来描述NS，并且VNFD必须定义使用TEST类型的VIM实例的端点。

如何扩展DUMMY VNFM AMQP：

VNF 参数

VNF需要的参数在VNFD中分为两种定义：

配置类参数：这类参数即key-value list，此类参数总可以从脚本中获得。

provides参数：这类参数在所有脚本中可见，区别是参数值只有在运行时才能得到。因此这类参数值只能由特定的VNFM来填充（从特定VNFM的实现类的方法中填充）

还有一些out-of-the-box参数，可以自动被加到VNFR里。这些参数包含floatingIP和hostname等
