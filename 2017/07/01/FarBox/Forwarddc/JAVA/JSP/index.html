JSP知识

web应用目录结构：

myweb
|–WEB-INF
| |–classes
| |–lib
| |–web.xml
xxx.jsp

web.xml

实际上每个web容器，如tomcat，也有自己的web.xml，作为其管理的所有web应用的全局配置
web.xml的metadata-complete熟悉为true时，不会加载注解配置的web组件

JSP原理

JSP在HTML里嵌入JAVA代码，每个JSP页面会被web容器（例如TOMCAT）转化成成一个继承HttpServelet的JAVA类称为servlet
servlet类实现了service方法响应请求，实际是将JSP页面输出为标准HTML页面返回给客户端的浏览器
编译该类生成CLASS文件，当servlet第一次被请求时，执行CLASS文件
JSP页面生成的servlet的java文件和class文件均存放在tomcat的work路径对应的web应用下

JSP编译指令

语法格式：<%@ 指令名 key=value … %>

page编译指令：
位于JSP页面顶端，定义一些JSP页面的属性。包括：import包、缓冲区、错误页面、MIME类型
错误页面的意思：服务端JSP抛出异常时，通过指定错误页面，会返回错误页面给客户端

include编译指令：
导入页面。效果是JSP的servlet代码里已经完全融入了被导入页面的servlet代码

动作指令

编译指令是在将JSP编译成servlet class时生效，动作指令是在执行class时，即运行时生效
语法：<jsp:命令名 … ><jsp:param … /></jsp:命令名> 或者 <jsp:命令名 … /> 前者可以给forward\include动作指令增加参数

forward动作指令：
将请求交给另一个JSP页面，携带用户的请求参数和forward指令附带的参数，对用户来说这种forward是透明的

include动作指令：
与编译指令区别是：include动作指令只导入JSP页面的body部分，被导入页面的编译指令也不会被导入

useBean动作指令：
在JSP中初始化一个JAVA实例，并且为该实例指定scope，scope包括：page、request、session、application
这里用到的JAVA对象，其编译后的CLASS文件放在WEB-INF/classes/路径下
可以看到，WEB应用的classes路径保存自身应用中使用到的javaBean，而tomcat的work保存的是用于响应客户端的class

JSP内置对象

JSP里的内置对象是Servlet里的实例，只是这些实例在Servlet的service方法里进行了默认初始化

application：
1、用于WEB应用内部多个JSP、Servlet之间共享数据。application在Servlet里对应ServletContext实例
2、获取WEB应用在web.xml里定义的全局配置参数。参数定义在元素里

config：
用于获取当前JSP或者Servlet里的配置参数
web.xml里配置的是，指定servlet-name\jsp-file\init-param 等参数，这些参数可以通过config获取
完成请求与处理的Servlet的映射
客户端请求的时候如果通过映射路径访问JSP，则config对象可以获取web.xml里的当前JSP配置参数，如果直接请求jsp页面，则无法获取
这里能感觉出来，其实JSP只是生成的Servlet的”草稿文件”

exception：
错误页面使用exception对象输出异常信息

pageContext：
用于set或get指定scope（page\request\session\application）的参数
用于获取其他内置对象

request：
用于获取GET\POST请求参数
用于set/get request范围的参数

response：
生成非字符响应

Servlet的方法

service：解析请求参数、处理请求、生成响应

tomcat目录结构：
{tomcat-dir}
|–bin : tomcat命令对应的二进制文件
|–conf : 所有配置
|–lib : tomcat服务器的核心类库、第三方类库
|–logs : 每次运行tomcat生成的日志
|–temp : 运行中生成的临时文件
|–webapps : 存放web应用
|–work : 保存tomcat运行过程中编译生成的class文件，每次启动tomcat会重新建立

tomcat部署web应用存在两种方式：
1、直接将WEB目录拷贝到webapps目录下
2、在conf/catalina/localhost/xx.xml中，配置web应用在本地的路径。这时web应用的路径是xml文件名

web应用数据源配置：
第一种部署方式，需要在web.xml里或者基于注解的方式配置数据库连接参数
第二种部署方式时，在xml文件里增加Resource元素，来配置datasource。jsp里可以通过JNDI获取数据库连接，JNDI就是在配置中代表这个数据源的名称
为了支持datasource，需要在tomcat的lib下或者在web应用的lib目录下保存JDBC驱动，区别是全部web应用和单独web应用可以使用JDBC驱动

jsp:useBean

该命令用于在JSP代码里使用JAVA对象。创建的JAVA类放在WEB-INF/src/路径下，注意如果要让JSP找到该类，必须将类打包，例如src/MyPkg/路径下才是你的JAVA文件。useBean命令的class参数指明该类，形式如“包名.类名”。

Jsp:getProperty 和jsp:setProperty调用该类的get\set方法，注意get\set方法的属性名和getProperty\setProperty指令的name值必须一致，大小写无关。

Application

JSP里application是一个map数据结构的内置变量。web应用下的所有JSP、servelet都可以访问application的内容。jsp的application对应servlet里的ServeletContext对象。application内置对象的k-V数据保存在web.xml在里

Config

也是JSP里的内置对象，可以访问单个JSP页面的配置信息，配置的信息保存在WEB-INF/web.xml里。web.xml里通过保存某个JSP页面的配置信息，但在JSP里不能通过config对象访问这些配置信息，而是需要在web.xml里指定该JSP页面的servlet映射路径URL（指向该JSP，可以通过访问它得到保存在web.xml里的JSP配置信息）之后，通过访问配置路径URL的方式才可以访问JSP的配置信息。

web.xml里指定JSP的servlet映射路径URL的方式：的key为config，value为servlet映射路径URL（具体到路径下的文件）。

Exception

也是JSP内置对象，但该对象只能用在errorPage。当一个JSP出现异常，如果指定了errorPage，则会跳到errorPage。为什么JSP自己不处理异常呢？因为所有JSP里的代码到Servelet里看，会发现都在一个try块里，所以不需要JSP处理异常，而Servelet的catch块就是判断是否存在errorPage，存在就执行errorPage。对了，JSP异常处理对JSP声明（在JSP里声明JAVA变量）无效，应该值声明的异常方法要自己处理异常吧。

Out

JSP内置管输出的对象，但所有用到out的地方都可以通过输出表达式<%= %>代替

pageContext

JSP内置对象，可以灵活的在四个scope（page\request\session\application）使用set\get设置key-value对。

Request

通过内置的request对象，获取客户端请求的请求头和参数。请求头由浏览器添加，参数则是服务器端开发人员控制。表单提交的客户端参数，必须是有name的参数才可以getValue。

表单里包含非英文的字符时，在获取前使用request.setCharacterEncoding指定编码方式来获取这些参数。但该方法只对post的参数有效。

服务器端处理表单数据的JSP页面如果使用forward指令转发给另一个页面，客户端请求的参数依旧可以通过request获取，而且处理表单的JSP页面还可以通过request.setAttribute生成服务器端的参数，传给forward到的JSP页面，这时参数的value可以是java对象。

Get\post请求

get请求：用户输入url访问网站或者提交表单且form method为默认值时，为get请求。get请求的参数key-value对附在url后面，形如url?key=value&key=value…… 。一般get请求数据量小。

post请求：提交表单的form method指明为post时，为post请求。数据量大，且参数保存在html header里，安全性高。

Response

需要生成非字符响应（例如响应一张图片）时，使用response内置对象响应客户端请求。out内置对象只能响应（输出）字符对象。

还可用于请求重定向，response的snedRedirect方法重定向到页面时，无法携带原请求的参数和request范围的key-value，与forward不同。

还可以用于向客户端写cookie。cookie用于服务器端记录客户端某些信息（用户名等）。服务器端通过request获得客户端的参数信息来创建cookie对象，再通过response.addCookie向客户端写cookie。之后服务器端可以通过request.getCookies得到cookie[]，每个cookie是一个key-value。

Session

保存C\S间一次session的key-value对。一次session指的是客户端链接到服务器到断开（关闭浏览器）的过程。session保存用户的会话信息（购物记录等）。

Servlet

地位

MVC规范后，用于C，即控制器。

基本概念

继承自HttpServlet，包含三个方法，init\service\destroy，一般情况下只需要override service方法响应客户端请求。

配置

写好了servlet（所有的JAVA类包括Servlet都放在web应用的/WEB-INF/SRC/路径下），需要编译成class，但编译完后，servlet仍然无法响应客户端，这时需要对servlet进行配置。配置分为两种，注解和web.xml。

注解：
在servlet代码里使用@WebServlet的方式配置servlet。

web.xml：
使用元素配置。两种方式都需要配置的是servlet的name（servlet-mapping时可以根据name，map到该servlet）和URL（客户端访问该servlet的URL）。两种配置方式不能同时使用。
还可以配置servlet参数initParam，访问servlet配置参数通过ServletConfig对象，也就是JSP内置对象config

Servlet实例化的时机

servlet存在两种实例化时机，一种是在客户端初次发起请求时，由web容器实例化该servlet。另一种是随着web容器启动就实例化（load-on-startup），这种servlet一般不负责响应客户端。具体指定哪种实例化时机，可以通过servlet配置来设定。

使用Servlet作为控制器

负责收集用户请求数据、将处理结果呈现给用户（调度）。处理过程通过调用M（Model，即JavaBean，JAVA对象，参见JSP指令useBean）来实现。所以，MVC模式可以理解为：M-Model-JavaBean，V-View-JSP，C-Controller-Servlet

JSP标签

为了不在JSP里使用蹩脚的JSP脚本，开发了自定义标签替代JSP脚本。

标签库taglib

标签库文件以tld(tag library defination)结尾，文件内容为xml。根元素是taglib定义标签库，taglib包含若干tag标签，每个tag对应一个JAVA类，该类是标签处理类，需extends SimpleTagSupport且override doTag方法，使用标签时调用的就是标签处理类的doTag方法。所有的JAVA类都放在WEB-INF/src路径下，tld标签库文件放在WEB-INF路径下。

JSP导入标签库

使用JSP编译指令<%@ taglib uri="uri" prefix="prefix"%>导入taglib标签库，需指定taglib-uri属性找到taglib，prefix属性用于使用标签时指定前缀表示标签所属标签库。

JSP使用标签

<prefix:tagname/>，prefix对应导入taglib指定的prefix，tagname为在tld定义tag时的name值。

带属性的标签：
如果标签的JAVA类有定义属性，必须提供属性的getter和setter方法。tld文件定义tag时，可以通过元素声明这些属性，attribute name与标签处理类的属性名对应。在JSP里使用标签时，可以根据tld里tag的attribute name给标签处理类的属性赋值。

带标签体tagContent的标签：
使用标签体，在JSP使用标签的时候就可以实现标签体代码，而非实现在标签处理类的doTag方法里，即实现了JSP和tag处理类搭配处理任务的作用，这在MVC模式很有意义。例如JSP标签体负责显示，而处理交给tag处理类。
定义tag时，指明标签体类型，例如scriptless表示标签体不能是JSP脚本，可以是静态HTML。JSP使用tag时，通过 <prefix:tagname …> yr tag-content </prefix:tagname> 添加标签体内容，在标签处理类里有对应的方法获取标签体内容。
