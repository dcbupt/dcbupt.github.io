环境搭建

该教程讲解在LINUX上安装稳定的二进制版本OpenBaton环境。

组件

该版本由以下几部分构成：

  • NFVO
    使用SPRING.IO框架
  • RABBITMQ
    作为消息系统
  • 测试版VIM驱动程序
    目的是可以不需要OPENSTACK就可以执行一个HELLOWORLD示例

以下部分为可选组件：

  • OPENSTACK VIM驱动
    用于在OPENSTACK上部署VNF
  • 通用VNFM
  • 故障管理系统
    支持检测和恢复VNF
  • 自动扩缩引擎
    根据性能需要自动创建或删除VNF
  • 网络分片引擎
  • MYSQL
    作为一种OPENBATON持久化的方式

安装OPENBATON

安装依赖：

为了简化安装过程我们提供了一个bootstrap脚本，该脚本会负责安装你想要的组件并配置他们来运行一个“HELLOWORLD”VNF。
为了执行bootstrap脚本，你需要安装curl
apt-get install curl

注：
我们假设你装在一个纯净的UBUNTU14.04上，其他情况下我们建议你一个个安装组件，你可以查看bootstrap项目中脚本执行的安装过程。

执行脚本：
sh <(curl -s http://get.openbaton.org/bootstrap) release

非常重要：

默认RABBITMQ装在了运行NFVO的主机上，特别注意安装的过程中你需要输入RABBITMQ的IP和端口，请确保该IP可以被外部组件访问（虚机或者你运行VNFM的主机），
否则你会遇到运行时错误。如果你把OPENBATON装在OPENSTACK管理的虚机上，这里你最好输入的是FLOATING IP

在安装过程中，你需要进行一些输入。例如选择是否安装通用VNFM或者其他组件，以及是否启用SSL

安装结束如果没有问题，你可以在LOCALHOST:8080上访问DASHBOARD，并且在/usr/lib/openbaton目录下可能会存在这些目录：

  • nfvo
    包含nfvo的jar包
  • vnfm/generic
    包含G-VNFM的jar包
  • fms
    包含故障管理系统的jar包
  • ase
    包含自动扩缩引擎的jar包
  • nse
    包含网络分片引擎的jar包
  • plugins
    包含OPENBATON的插件。默认测试版VIM驱动插件已经安装，因此该驱动的JAR包会放在该目录里。而且安装过程中你决定安装适配OPENSTACK的驱动插件，
    该插件的JAR包也会放到该目录里。
  • systemd
    包含OPENBATON的系统配置文件以及服务管理器

根据你安装的组件不同，在/usr/bin目录下可能会包含：

  • openbaton-nfvo
    NFVO执行器
  • openbaton-vnfm-generic
    G-VNFM执行器
  • openbaton-fms
    FMS执行器
  • openbaton-ase
    ASE执行器
  • openbaton-nse
    NSE执行器

HELLOWORLD教程

该教程讲解如何部署一个由仿制的VNFs构成的NSR，目的是验证NFVO部署成功。它不会创建任何虚机也不会真正部署一个网络服务。
你也可以用TOSCA来做该教程。

依赖：

  • NFVO
  • 测试版VIM驱动
  • DUMMY VNFM

准备：

运行NFVO和DUMMY VNFM

如果你使用bootstrap脚本安装NFVO，测试版VIM驱动默认已经安装了

上载VIM实例：

上载一个测试版VIM实例到NFVO，测试类VIM实例会确保NFVO使用测试版VIM驱动器来部署NS

你需要上载如下的测试版VIM实例

使用DASHBOARD：

登录使用admin用户名和openbaton密码（默认）

MANAGE POPS -> POP INSTANCES -> REGISTER VIM，填入VIM实例的JSON描述

使用CLI：

如果你想使用CLI来上载VIM实例，使用下面的命令：
$./openbaton.sh VimInstance-create vim-instance.json

上载NSD：

NSD的上载也可以通过DASHBOARD或者CLI

使用DASHBOARD：

Catalogue -> NS Descriptors -> Upload NSD

使用CLI：
$./openbaton.sh NetworkServiceDescriptor-create tutorial-iperf-NSD.json

部署NSD

一旦你完成了NSD的上载，你就可以部署NSD

这会创建一个NSR，但因为使用的是DUMMY VNFM AMQP，所以不会创建虚机

使用DASHBOARD：

CATALOGUE – NS DESCRIPTORS – ACTION – LAUNCH

使用CLI：
$./openbaton.sh NetworkServiceRecord-create bef3c744-b02a-48f1-9d14-9ede7847bc57 vimmap.json keypair.json conf.json

第一个参数：NSD的ID
后面的参数：额外的配置文件，如果你不需要，也必须创建这样三个文件，vimmap和conf文件填写{}，keypair填写[]

结论

当NFVO和DUMMY VNFM完成了它们的工作，部署的NSR的状态会变为ACTIVE，但没有真正的虚机被创建也没有真正的网络服务被部署

测试版VIM驱动保证NFVO认为所有需要的资源都被分配了而且VIM实例也被创建了。DUMMY VNFM使得NFVO相信虚机被创建了并且在其上执行了生命周期事件脚本。

配置NFVO

除非你想调整安装OPENBATON，否则使用默认的bootstrap脚本安装是不需要配置NFVO的。

如果你配置了NFVO，你同样也要修改G-VNFM的参数配置

NFVO属性概览

在安装完成后，NFVO的配置参数位于
/etc/openbaton/openbaton-nfvo.properties

这是一个用于配置SPRING环境和NFVO的属性文件，既然组件是基于SPRING框架则一些参数是继承自SPRING的

切记无论下面讲到的任何参数被修改了，你都需要重启NFVO

修改NFVO通用属性

默认情况下，RABBITMQ装在NFVO主机上，注意当你的任何组件运行在其他主机上时，你要保证该主机依然可以访问RABBITMQ

其实NFVO也可以在其他的主机上运行，只需要修改配置文件内容：
旧：nfvo.rabbit.brokerIp = localhost
新：nfvo.rabbit.brokerIp = <the rabbitmq broker ip>

我理解，OPENBATON默认安装是装了NFVO和消息系统RABBITMQ的（所以NFVO和RABBITMQ的IP一样）
如果你当前主机还装了其他组件（例如VNFM），且你希望NFVO跑在另一台主机上，那么你就需要按照上面的方式修改NFVO的配置文件
