系统学习Spring源码-Spring容器

前言

DefaultListableBeanFactory是Spring的bean工厂，实现IoC控制反转，集中管理bean之间的依赖，实现依赖注入。但Spring框架不仅有bean工厂，还有许多其他特性，因此Spring提供核心接口ApplicationContext来描述一个可配置的Spring容器，一个ApplicationContext需要具备如下能力：

  • 可配置的beanFactory
  • 支持多种Spring配置方式
  • 支持应用事件广播和消费

ApplicationContext架构

基础类

ApplicationContext

  • 顶层接口，用来指代一个Spring容器，getAutowireCapableBeanFactory方法用来获取一个可配置可依赖注入的bean工厂，可见Spring容器的核心就是包了一个基础的bean工厂

ConfigurableApplicationContext

  • 继承ApplicationContext，描述一个可配置的Spring容器，提供配置方法，例如添加bfppaddBeanFactoryPostProcessor、添加应用事件监听器addApplicationListener
    • 类比ConfigurableBeanFactory，描述的是可配置的bean工厂，实现对bean加载过程的定制，例如给bean工厂配置bpp。对于Spring容器而言，它的可配置体现在对bean工厂本身的定制化，因此提供的扩展是bean工厂处理器bfpp。需要说明的是，bfpp虽然定制的是bean工厂，但有些bfpp也是调用的bean工厂的配置接口，本质上还是对bean加载过程的定制
  • 定义核心方法refresh，该方法在ApplicationContext的构造函数里调用来刷新（初始化）Spring容器

核心类

AbstractApplicationContext

  • ConfigurableApplicationContext的基础实现，实现了refresh方法，框定了Spring容器刷新的主流程，该方法的实现后面会详细介绍
    • 提供模板方法refreshBeanFactory，子类实现来创建beanFactory

AbstractRefreshableApplicationContext

  • AbstractApplicationContext的子类，实现了refreshBeanFactory方法，new一个bean工厂DefaultListableBeanFactory提供给它的子类去解析和注册beanDefinition
    • 读取Spring容器的配置属性来定制bean工厂，包括是否支持循环依赖、是否支持相同beanName的beanDefinition覆盖
    • 提供模板方法loadBeanDefinitions，由子类负责加载bean配置定义、解析生成beanDefinition并注册到beanFactory
      • AnnotationConfigWebApplicationContext实现loadBeanDefinitions，将注解配置的beanDefinition注册到beanFactory
      • AbstractXmlApplicationContext实现loadBeanDefinitions，将xml配置的beanDefinition注册到beanFactory
  • GenericApplicationContext和AbstractRefreshableApplicationContext平级，也实现了refreshBeanFactory方法，但可以理解为一个空实现，实际是在无参构造函数里new一个bean工厂DefaultListableBeanFactory，子类在自己的构造函数里完成特定配置方式加载和注册beanDefinition
    • AnnotationConfigApplicationContext是注解方式的spring容器
    • GenericXmlApplicationContext是xml方式的spring容器

AbstractRefreshableConfigApplicationContext

  • AbstractRefreshableApplicationContext的子类，提供对bean配置文件路径解析的能力
    • 将占位符替换为实际的系统环境变量

AbstractXmlApplicationContext

  • AbstractRefreshableConfigApplicationContext的子类，实现方法loadBeanDefinitions，完成xml方式beanDefinition解析和注册
    • 使用辅助类XmlBeanDefinitionReader完成xml配置文件的解析和beanDefinition的注册
    • 提供模板方法getConfigResources，由子类负责加载Spring配置文件
  • AnnotationConfigWebApplicationContext和AbstractXmlApplicationContext平级，也实现了loadBeanDefinitions，完成注解方式的beanDefinition解析和注册

ClassPathXmlApplicationContext

  • xml配置方式的Spring容器
  • AbstractXmlApplicationContext的子类，实现了getConfigResources方法，使用辅助类ClassPathResource加载xml配置文件

总结

ApplicationContext架构的UML类图：

综上，ApplicationContext架构主要使用了适配器模式、模板模式

  • ApplicationContext通过在内部持有一个DefaultListableBeanFactory，适配了BeanFactory接口
  • Spring容器的刷新主流程在上层抽象类的refresh方法中实现。同时定义创建bean工厂的模板方法refreshBeanFactory，有两套bean工厂创建流程，分别由子类AbstractRefreshableApplicationContext和GenericApplicationContext实现
    • AbstractRefreshableApplicationContext类和GenericApplicationContext类都有自己的子类，分别支持xml方式和注解方式的beanDefinition注册

刷新Spring容器

刷新（初始化）Spring容器的入口在org.springframework.context.support.AbstractApplicationContext#refresh，构造函数里调用

下面详述refresh方法的实现逻辑

定制系统变量

回调模板方法initPropertySources，由三方实现，可以添加系统变量、设置哪些系统变量是必须的（不存在直接抛异常）

测试代码：applicationcontext.RequiredPropertyTest

构建DefaultListableBeanFactory

构建Spring容器的DefaultListableBeanFactory，提供refreshBeanFactory模板方法，由子类实现，创建一个已经完成beanDefinition解析和注册的bean工厂

显式配置DefaultListableBeanFactory

注册SPEL解析器

显式注册解析SpEL的StandardBeanExpressionResolver，该resolver会在bean属性赋值前被调用来解析配置属性的SpEL表达式

注册属性编辑注册器

显式注册ResourceEditorRegistrar，注册器会在Bean加载阶段被调用，给BeanWrapper注册一些资源类型的PropertyEditor

  • 如果用户想添加自定义的PropertyEditor，可以简单覆盖PropertyEditorSupport的setAsText方法来自定义一个PropertyEditor，再创建它的注册器PropertyEditorRegistrar，然后配置一个CustomEditorConfigurerBean，在propertyEditorRegistrars属性中加入自定义的Registrar。CustomEditorConfigurer是bfpp，在beanFactory后置处理中将PropertyEditorRegistrar注册到BeanFactory

注册ApplicationContextAwareProcessor

显式注册bpp：ApplicationContextAwareProcessor，它的作用是bean前置初始化时，为bean自动注入Aware方式指定的Spring容器组件

注册ApplicationListenerDetector

显式注册bpp：ApplicationListenerDetector，作用是对于类型为ApplicationListener子类的bean，后置初始化处理时注册到ApplicationContext。这些应用事件监听器最终会被注册到事件组播器ApplicationEventMulticaster，Spring容器关闭时，ApplicationListenerDetector会在bean销毁前置回调时，从ApplicationEventMulticaster里移除

系统环境变量以bean的形式注册到BeanFactory

回调三方对bean工厂的定制接口

提供模板方法postProcessBeanFactory，由AbstractApplicationContext的子类（第三方实现类）覆写，定制beanFactory

private void startBeans(boolean autoStartupOnly) {
    Map<String, Lifecycle> lifecycleBeans = getLifecycleBeans();
    Map<Integer, LifecycleGroup> phases = new HashMap<>();
    lifecycleBeans.forEach((beanName, bean) -> {
        if (!autoStartupOnly || (bean instanceof SmartLifecycle && ((SmartLifecycle) bean).isAutoStartup())) {
            int phase = getPhase(bean);
            LifecycleGroup group = phases.get(phase);
            if (group == null) {
                group = new LifecycleGroup(phase, this.timeoutPerShutdownPhase, lifecycleBeans, autoStartupOnly);
                phases.put(phase, group);
            }
            group.add(beanName, bean);
        }
    });
    if (!phases.isEmpty()) {
        List<Integer> keys = new ArrayList<>(phases.keySet());
        Collections.sort(keys);
        for (Integer key : keys) {
            phases.get(key).start();
        }
    }
}

总结

综上，可以总结出ApplicationContext具备这些定制和扩展能力：

  • 支持SPEL
    • 显式注册表达式解析器到beanFactory
  • 支持自定义bean属性编辑器
    • 通过bfpp后置处理时把属性编辑注册器注册到beanFactory
  • 支持自定义BeanFactoryPostProcessor
    • 显式注册或配置方式自定义bfpp
  • 支持自定义BeanPostProcessor
    • 配置方式发现自定义bpp
  • 支持自定义发布和监听应用事件
    • 配置方式自定义事件组播器，否则Spring容器提供默认的组播器
    • 显式注册或配置方式自定义bfpp
  • 支持Spring容器在启动或停止时刻回调LifeCycle bean对应的生命周期方法
