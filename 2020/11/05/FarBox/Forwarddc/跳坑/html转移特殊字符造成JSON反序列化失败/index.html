html转移特殊字符造成JSON反序列化失败

背景

从数据库查到的json结构的IM聊天记录做反序列化失败

通过下面的demo简单复现问题：

首先，json在序列化时，会在特殊字符前增加反斜杠转义符\，在反序列化时会去掉反斜杠

然后说下消息存储和读取的服务端逻辑：

  • 端上将消息按{"text":"xxx"}的格式json序列化后，调服务端接口发送消息。服务端拿到消息后，按约定的格式反序列化后，单独取出text的value做html特殊字符转义，然后再json序列化落库
  • 读取消息接口，从数据库里取出序列化的json串后，先对整个json串做html特殊字符反转义，然后再反序列化得到消息对象

这套流程应对绝大多数文本消息是没问题的，但当文本消息中出现英文双引号"时，就会在读取消息接口进行反序列化操作时报错

具体原因是

  • 存储消息时，服务端先做了反序列化，这时再读取text字段时，"前面不会加\转义符，因此html转义后也不带\
  • 读取消息时，做html反转义输出的json串的text字段的值，"前自然也就没有\，所以不满足json串的结构，当然反序列化报错

当然，如果在读取消息接口里，查库后先反序列化，再对text的value单独做html反转义就不会有问题，因为这正好和写库前的处理是完全逆向的。之所以读取时破坏了这种逆向，是因为有些服务端不感知IM消息结构，因此无法单独取出text字段的value值，只能对整个json做html反转义，再反序列化。

所以，对于有html转义存储的场景，尽量对整个String做html反转义，不然一旦需要json序列化，就可能出现这种json反序列化失败问题
